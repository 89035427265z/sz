User Stories — Screen 8: Сдача задания
Acceptance Criteria · Бизнес-ценность · Технические ограничения
US-801
Как исполнитель, я хочу подтвердить своё местонахождение через GPS, чтобы система убедилась, что я нахожусь в нужном магазине перед сдачей задания.

Критерии приёмки

  • Разрешение на геолокацию запрашивается при первом открытии Screen 8 (expo-location)
  • GPS фиксируется с точностью accuracy ≤ 50м
  • Пользователь должен находиться в радиусе 200м от координат магазина
  • При accuracy > 50м — предупреждение «Слабый GPS», переход к запасной логике (Wi-Fi / manual)
  • GPS-координаты сохраняются в EXIF-метаданных каждой фотографии
  • Время GPS-фиксации: не ранее 5 минут до отправки отчёта
  • Метод верификации логируется: location_method: enum(gps|wifi|manual)
US-802
Как исполнитель, я хочу загрузить минимум 3 фото через камеру, чтобы подтвердить факт и качество выполнения задания.

Критерии приёмки

  • Минимум 3 фото, максимум 10
  • Фото снимаются только через камеру приложения — галерея заблокирована (expo-camera)
  • EXIF DateTimeOriginal каждого фото: не ранее 30 минут до момента отправки
  • Уникальность: perceptual hash расстояние между любыми двумя фото > 10
  • Минимальное разрешение: 800×600px; максимум на клиенте 5 МБ → сжимается до 1 МБ перед upload
  • Удаление фото: возможно до отправки, невозможно после
  • Хранение в MinIO по пути: {task_id}/photo_{n}_{timestamp}.jpg
  • Прогресс-индикатор загрузки для каждого фото (chunked upload)
US-803
Как исполнитель, я хочу добавить текстовый комментарий, чтобы объяснить особые обстоятельства выполнения.

Критерии приёмки

  • Поле опциональное, максимум 500 символов
  • Счётчик символов обновляется в реальном времени: «243 / 500»
  • При превышении лимита — поле подсвечивается красным, отправка блокируется
  • Ненормативная лексика — предупреждение, без блокировки
US-804
Как исполнитель, я хочу получить подтверждение успешной отправки, чтобы знать, что задание принято и оплата будет начислена.

Критерии приёмки

  • Success screen появляется ≤ 3 секунд после подтверждения сервером
  • На экране: сумма к начислению, ожидаемая дата выплаты
  • Статус задания изменяется: in_progress → pending_review
  • Push-уведомление: «Отчёт по заданию #ID принят. Сумма: X ₽»
  • При ошибке сервера: кнопка «Повторить» + черновик сохраняется локально
  • Анимация конфетти при успехе (отключается при Reduce Motion)

User Stories — Screen 4: Регистрация
SMS · OCR паспорта · ФНС · ПЭП договор ГПХ
📌
Онбординг состоит из 4 шагов: SMS → OCR паспорта → ФНС проверка → Подписание договора ГПХ. Каждый шаг — отдельная User Story с независимыми критериями приёмки.
US-401
Как новый пользователь, я хочу зарегистрироваться по номему телефона, чтобы создать защищённый аккаунт в системе KARI.

Критерии приёмки

  • Маска ввода: +7 (XXX) XXX-XX-XX
  • OTP: 6 цифр, срок действия 5 минут
  • Максимум 3 попытки ввода OTP → блокировка на 15 минут с обратным отсчётом
  • Кнопка «Повторить SMS» активна через 60 секунд (таймер с отсчётом)
  • Доставка SMS ≤ 60 сек через SMSC.ru
  • Auto-read OTP из SMS на Android (SMS Retriever API)
US-402
Как новый пользователь, я хочу отсканировать паспорт камерой, чтобы данные заполнились автоматически и я не вводил вручную.

Критерии приёмки

  • Съёмка только через камеру приложения (expo-camera, без галереи)
  • Распознаётся: ФИО, дата рождения, серия/номер, дата выдачи, код подразделения, адрес прописки
  • Успешность распознавания ≥ 90% при нормальном освещении
  • Таймаут OCR > 10 сек → предложить ручной ввод
  • Фото паспорта НЕ сохраняется на сервере после распознавания (152-ФЗ)
  • Данные шифруются AES-256-GCM через crypto_service.py перед записью в БД
  • После OCR — все поля редактируемы для ручной правки
US-403
Как новый пользователь, я хочу пройти проверку статуса самозанятого через ФНС, чтобы подтвердить право на получение заданий.

Критерии приёмки

  • Проверка: GET /api/fns/check-inn — ответ ≤ 5 сек
  • Статус «активный самозанятый» → следующий шаг
  • Статус «не самозанятый» → инструкция с deeplink на приложение «Мой налог»
  • API ФНС недоступен → временный доступ с меткой fns_pending, повторная проверка через 24ч
  • Бывший сотрудник KARI (< 2 лет) → автоблокировка + уведомление HR (стоп-лист: former_employee)
US-404
Как новый пользователь, я хочу подписать договор ГПХ электронно, чтобы официально начать получать задания и выплаты от KARI.

Критерии приёмки

  • Договор генерируется из шаблона через reportlab (PDF), данные подставляются автоматически
  • Подпись через ПЭП — SMS-код подтверждения (тот же механизм, что и авторизация)
  • Договор доступен для скачивания в разделе «Документы» в любое время
  • HR получает push-уведомление о новом исполнителе после подписания
  • До подписания договора — задания не отображаются (заглушка с объяснением)

Edge Case: GPS «плавает» в торговом центре
Каскадная верификация местоположения — 3 уровня
⚠️
Проблема: В крытых ТЦ GPS accuracy достигает 200–500м из-за многоэтажных конструкций и помех. Простая проверка радиуса даёт ложные блокировки до 40% заданий в торговых центрах.
🔁 Каскадная логика верификации
L1
GPS (Primary)
accuracy ≤ 50м AND расстояние до магазина ≤ 200м → ✅ PASS. Записывает location_method: "gps"
L2
Wi-Fi геолокация (Fallback)
Если GPS accuracy > 50м: сканировать BSSID точек. Сравнить с реестром BSSID магазина (задаётся HR при создании задания). Совпадение ≥ 2 из списка → ✅ PASS. location_method: "wifi"
L3
Ручное подтверждение (Last resort)
Если Wi-Fi не совпал: показать сообщение «GPS слабый — сфотографируйте вывеску магазина». Обязательное «proof photo» с вывеской. Статус задания = manual_review. Push директору магазина для верификации.
LOG
Аудит-лог (все уровни)
В audit_log записывается: { gps_accuracy, wifi_bssids_seen, wifi_match_count, final_method, timestamp }

Edge Case: Потеря интернета при отправке фотоотчёта
Offline-first · Chunked upload · Resumable · Idempotency
📡
Проблема: 3–10 фото (~5–15 МБ) отправляются из магазина в зоне слабого 4G. При обрыве без защиты — исполнитель теряет отчёт и не получает оплату.
// ФАЗА 1 — Сжатие до загрузки (клиент, всегда) Каждое фото: resize 800×600, JPEG quality=75 → ~150–200 КБ 3–10 фото итого: ≤ 2 МБ // ФАЗА 2 — Chunked Upload (resumable) POST /api/tasks/{id}/report/session ← { session_id, chunk_size: 256KB, total_chunks: N } PUT /api/tasks/{id}/report/chunks/{n} ← { chunk: n, received: true } // подтверждение каждого chunk POST /api/tasks/{id}/report/commit ← сервер склеивает файл → сохраняет в MinIO // ФАЗА 3 — Offline Queue (при потере соединения) AsyncStorage.setItem("draft_report_{task_id}", JSON.stringify({ photos: [base64...], gps: { lat, lon, accuracy, method }, comment: "...", created_at: "2026-03-11T10:00:00Z", uploaded_chunks: [0, 1, 2] // уже загружены — не повторять })) // При восстановлении соединения: NetInfo.addEventListener("connected") → → возобновление с последнего подтверждённого chunk → Snackbar: "Соединение восстановлено. Отправка: 3/5 фото..." → Таймаут черновика: 24ч предупреждение, 48ч удаление // Защита от двойной отправки: idempotency_key = sha256(task_id + user_id + date) // Сервер при повторном запросе → HTTP 200 "already_accepted"

API Design — Валидация отчёта на бэкенде
Pydantic-модели · 7 проверок · Fail-fast · Idempotency
🛡️
Проверки выполняются в строгом порядке — fail-fast: первая критическая ошибка прерывает цепочку. Некритические ошибки (anti-fraud L6–L7) собираются в массив errors[] и возвращаются все вместе.
# backend/app/api/tasks.py class PhotoMetadata(BaseModel): filename: str size_bytes: int = Field(gt=0, lt=5_242_880) # max 5 MB exif_timestamp: datetime gps_lat: Optional[float] = Field(None, ge=-90, le=90) gps_lon: Optional[float] = Field(None, ge=-180, le=180) phash: str # 64 hex-символа class SubmitReportRequest(BaseModel): task_id: str = Field(regex=r'^task_[a-z0-9]{8}$') session_id: str gps_lat: float = Field(ge=-90, le=90) gps_lon: float = Field(ge=-180, le=180) gps_accuracy: float = Field(gt=0, lt=10_000) location_method: LocationMethod # enum: gps | wifi | manual photos: List[PhotoMetadata] = Field(min_items=3, max_items=10) comment: Optional[str] = Field(None, max_length=500) idempotency_key: str = Field(regex=r'^[a-f0-9]{64}$') # sha256 async def validate_report(req, task, executor): # 1. Задание в нужном статусе (fail-fast) if task.status != TaskStatus.IN_PROGRESS: raise HTTPException(409, "TASK_NOT_IN_PROGRESS") if task.executor_id != executor.id: raise HTTPException(403, "NOT_YOUR_TASK") # 2. Дедлайн не истёк (fail-fast) if datetime.utcnow() > task.deadline_end: raise HTTPException(422, "DEADLINE_EXPIRED") # 3. Idempotency — защита от двойной отправки if await redis.get(f"report:{req.idempotency_key}"): return {"status": "already_accepted", "task_id": req.task_id} errors = [] # 4. GPS-радиус (только при gps-методе с хорошей точностью) if req.location_method == LocationMethod.GPS: dist = haversine(req.gps_lat, req.gps_lon, task.store_lat, task.store_lon) if dist > 200 and req.gps_accuracy < 50: errors.append({"field": "gps", "code": "OUT_OF_RANGE", "distance_m": round(dist)}) # 5. Временны́е метки фото (≤ 30 мин) now = datetime.utcnow() for i, photo in enumerate(req.photos): age = (now - photo.exif_timestamp).seconds / 60 if age > 30: errors.append({"field": f"photos[{i}]", "code": "PHOTO_TOO_OLD", "age_min": round(age)}) # 6. Уникальность (perceptual hash) hashes = [p.phash for p in req.photos] for i in range(len(hashes)): for j in range(i+1, len(hashes)): if hamming_distance(hashes[i], hashes[j]) < 10: errors.append({"field": f"photos[{i}]", "code": "DUPLICATE_PHOTO", "similar_to": j}) # 7. GPS из EXIF фото ≤ 500м от магазина for i, photo in enumerate(req.photos): if photo.gps_lat and photo.gps_lon: dist = haversine(photo.gps_lat, photo.gps_lon, task.store_lat, task.store_lon) if dist > 500: errors.append({"field": f"photos[{i}]", "code": "PHOTO_GPS_MISMATCH", "distance_m": round(dist)}) if errors: raise HTTPException(422, detail={"errors": errors}) await redis.setex(f"report:{req.idempotency_key}", 86400, "1") return True
